<objective>
## Goal
Add visible connection status indicator, error boundaries for API failures, and polish the WebSocket reconnection UX — so users always know when the dashboard is live vs disconnected.

## Purpose
The frontend currently reconnects silently with exponential backoff but gives no visual feedback. Users can't tell if data is stale. This plan adds a connection status banner, handles the server-side ping messages from Plan 07-01, and adds error handling for failed API fetches on the detail page.

## Output
- Modified `frontend/src/lib/stores/websocket.svelte.ts` — handle ping messages, expose reconnecting state
- New `frontend/src/lib/components/ConnectionStatus.svelte` — connection status banner
- Modified `frontend/src/routes/+layout.svelte` — mount ConnectionStatus globally
</objective>

```gherkin

When the WebSocket connection is lost
Then a subtle banner appears indicating "Reconnecting..." with the current retry state
```

## AC-2: Ping/Pong Handling
```gherkin
Given the WebSocket store receives a message with type "ping"
When the message is processed
Then it is silently acknowledged (no state update, no error) and the connection stays alive
```

## AC-3: Error Boundary for Stale/Failed Data
```gherkin
Given the dashboard has lost its WebSocket connection
When the user views project cards or the detail page

```
