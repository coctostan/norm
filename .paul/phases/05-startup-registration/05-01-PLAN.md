<objective>
## Goal
Create unified startup scripts and implement config file persistence so NORM can be launched with a single command and registered projects survive restarts.

## Purpose
NORM currently requires manually starting backend and frontend separately, and project registrations are lost if the database is wiped. Config persistence and a startup script are the first steps to making NORM a daily-use tool.

## Output
- `scripts/dev.sh` — single command starts backend + frontend in dev mode
- `scripts/start.sh` — production mode (backend serves built frontend)
- Config file integration — norm.yaml read on startup, written on project add/remove
</objective>

  <files>

  </files>
  <action>
    1. In backend/config.py:
       - Add `load_config()` function that reads norm.yaml (using pyyaml, already a dependency)
       - Returns dict with `projects` list (each item has `name` and `path`)
       - Returns empty projects list if file doesn't exist or is empty
       - Add `save_config()` function that writes current projects to norm.yaml
       - Reads current projects from database, writes to norm.yaml

    2. In backend/main.py lifespan:
       - After init_db(), call load_config()
       - For each project in config, auto-register if not already in DB
       - Use existing database functions (not HTTP endpoints) to check/add

    3. In backend/registry.py:
       - After successful POST /api/projects/ (create), call save_config()
       - After successful DELETE /api/projects/ (delete), call save_config()
       - Import save_config from config module

    4. Update norm.yaml to have proper structure:
       ```yaml
       projects:
         # - name: "my-project"
         #   path: "/path/to/project"
       settings:
         port: 8000
         db_path: "norm.db"
       ```

    Avoid: Breaking existing registration flow — config persistence is additive.
    Avoid: Complex config schema — keep it simple YAML.
  </action>
  <verify>

    3. Check GET /api/projects/ shows the auto-registered project

    5. DELETE a project via API, verify norm.yaml updated

<task type="auto">
  <name>Task 2: Startup scripts (dev + production)</name>
  <files>
    scripts/dev.sh,
    scripts/start.sh
  </files>
  <action>
    1. Create scripts/dev.sh:
       - #!/usr/bin/env bash
       - set -e
       - Resolve project root (script location based)
       - Check .venv exists, warn if not: "Run: python3 -m venv .venv && pip install -e '.[dev]'"
       - Check frontend/node_modules exists, warn if not: "Run: cd frontend && npm install"
       - Start backend with uvicorn (--reload for dev) as background process
       - Start frontend with npm run dev as background process
       - Trap SIGINT/SIGTERM to kill both processes
       - Print URLs: Backend http://localhost:8000, Dashboard http://localhost:5173
       - Wait for both processes

    2. Create scripts/start.sh:
       - #!/usr/bin/env bash
       - set -e
       - Check .venv exists
       - Check frontend/build/ exists, warn if not: "Run: cd frontend && npm run build"
       - Start backend with uvicorn (no --reload, production)
       - Print URL: Dashboard http://localhost:8000
       - Note: In production, frontend is pre-built static files served separately or via proxy

    3. Make both executable: chmod +x scripts/dev.sh scripts/start.sh

    Avoid: Auto-installing dependencies (let user control their environment).
    Avoid: Complex process management (no supervisord, just trap + wait).
  </action>
  <verify>
    1. Run ./scripts/dev.sh — both backend and frontend start
    2. Ctrl+C stops both processes cleanly
    3. Run ./scripts/start.sh — backend starts in production mode
  </verify>
  <done>AC-1, AC-2 satisfied: Startup scripts work for dev and production modes</done>
</task>
