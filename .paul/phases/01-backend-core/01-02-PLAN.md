<objective>
## Goal
Parse PALS state files (STATE.md, ROADMAP.md, PROJECT.md) from registered project paths into structured Python data, cache parsed state in SQLite, and expose sync/state endpoints.

## Purpose
NORM's core value is showing project state at a glance. Before the frontend can render anything meaningful, the backend must be able to read `.paul/` markdown files, extract structured data (phase, loop position, progress, blockers), and cache it for fast API access.

## Output
- `backend/parser.py` — markdown parsing functions for STATE.md, ROADMAP.md, PROJECT.md
- `backend/parser_models.py` — Pydantic models for parsed state
- Updated `backend/database.py` — new cache tables (project_state, project_phases)
- Updated `backend/models.py` — CRUD for cache tables
- Updated `backend/schemas.py` — API response models for project state
- Updated `backend/registry.py` — sync endpoint + state endpoint
- `tests/test_parser.py` — parser unit tests
</objective>

<acceptance_criteria>

## AC-1: STATE.md Parsing
```gherkin
Given a valid .paul/STATE.md file from a registered project
When the parser processes it
Then it returns structured data including: milestone name, phase number/total, plan identifier, status, loop position (plan/apply/unify as booleans), progress percentages, last activity timestamp, and any active blockers
```

## AC-2: ROADMAP.md Parsing
```gherkin
Given a valid .paul/ROADMAP.md file from a registered project
When the parser processes it
Then it returns structured data including: milestone name/version/status, and a list of phases each with number, name, plan count, status, and completion date if available
```

## AC-3: PROJECT.md Parsing
```gherkin
Given a valid .paul/PROJECT.md file from a registered project
When the parser processes it
Then it returns structured data including: project name, description, version, status string, and last updated date
```

## AC-4: State Caching in SQLite
```gherkin
Given parsed state data from a registered project
When a sync is triggered (via API call)
Then the parsed state is stored/updated in SQLite cache tables (project_state, project_phases) and the project's last_synced timestamp is updated
```

## AC-5: Sync and State API Endpoints
```gherkin
Given a registered project with a valid .paul/ directory
When POST /api/projects/{id}/sync is called
Then the project's state files are parsed, cached in SQLite, and the endpoint returns the parsed state

Given a project with cached state
When GET /api/projects/{id}/state is called
Then the cached state is returned as structured JSON
```

## AC-6: Parser Handles Missing/Malformed Files Gracefully
```gherkin
Given a registered project where one or more .paul/ files are missing or malformed
When the parser processes the project
Then it returns partial results for files that exist and null/default values for missing ones, without raising exceptions
```

<task type="auto">
  <name>Task 1: Create markdown parser and Pydantic models</name>
  <files>backend/parser.py, backend/parser_models.py</files>
  <action>
    Create `backend/parser_models.py` with Pydantic models:
    - `LoopPosition`: plan (bool), apply (bool), unify (bool), description (str)
    - `PhaseProgress`: number (int), name (str), plan_count (int | None), status (str), completed_date (str | None)
    - `ParsedState`: milestone (str), phase_number (int), phase_total (int), phase_name (str), plan (str), status (str), loop (LoopPosition), progress_milestone (float | None), progress_phase (float | None), last_activity (str | None), blockers (list[str]), decisions (list[dict])
    - `ParsedRoadmap`: milestone_name (str), milestone_version (str), milestone_status (str), phases (list[PhaseProgress])
    - `ParsedProject`: name (str), description (str), version (str), status (str), last_updated (str | None)
    - `ProjectFullState`: state (ParsedState | None), roadmap (ParsedRoadmap | None), project (ParsedProject | None), synced_at (str)

    Create `backend/parser.py` with functions:
    - `parse_state_md(content: str) -> ParsedState`: Parse STATE.md content
      - Extract "Milestone:" line → milestone name
      - Extract "Phase: N of M (Name)" → phase_number, phase_total, phase_name
      - Extract "Plan:" line → plan identifier
      - Extract "Status:" line → status
      - Extract loop position from the code block (✓ = True, ○ = False)
      - Extract progress percentages from progress lines
      - Extract "Last activity:" line
      - Extract blockers from "### Blockers/Concerns" section
      - Extract decisions from "### Decisions" table
    - `parse_roadmap_md(content: str) -> ParsedRoadmap`: Parse ROADMAP.md content
      - Extract milestone name, version from "**v0.1 Foundation** (v0.1.0)" pattern
      - Extract milestone status from "Status:" line
      - Extract phases from the phases table (| Phase | Name | Plans | Status | Completed |)
    - `parse_project_md(content: str) -> ParsedProject`: Parse PROJECT.md content
      - Extract project name from first H1 heading (before the " — ")
      - Extract description from text after " — " in H1
      - Extract version from "| Version |" table row
      - Extract status from "| Status |" table row
      - Extract last updated from footer or "| Last Updated |" row
    - `parse_project_state(project_path: str) -> ProjectFullState`: Orchestrator
      - Read STATE.md, ROADMAP.md, PROJECT.md from {project_path}/.paul/
      - Call individual parsers, catch exceptions per-file (return None for failures)
      - Return ProjectFullState with all parsed data

    Use regex and string splitting for parsing — no external markdown libraries needed.
    These are structured markdown files with consistent formatting conventions.
    Avoid: over-engineering with full AST parsing — simple regex/split is sufficient and faster.
  </action>
  <verify>
    Run: python -m pytest tests/test_parser.py -v
    All parser tests pass (created in Task 2 verify step uses these)
  </verify>
  <done>AC-1, AC-2, AC-3, AC-6 satisfied: All three file types parse correctly, missing files return None gracefully</done>

    **Update `backend/database.py`:**
    Add two new tables to _SCHEMA:
    - `project_state`: project_id (FK → projects.id, UNIQUE), milestone (TEXT), phase_number (INT), phase_total (INT), phase_name (TEXT), plan (TEXT), status (TEXT), loop_plan (BOOL), loop_apply (BOOL), loop_unify (BOOL), progress_milestone (REAL), progress_phase (REAL), last_activity (TEXT), blockers_json (TEXT — JSON array), decisions_json (TEXT — JSON array), project_name (TEXT), project_description (TEXT), project_version (TEXT), project_status (TEXT), synced_at (TIMESTAMP)
    - `project_phases`: id (INTEGER PK), project_id (FK → projects.id), phase_number (INT), name (TEXT), plan_count (INT), status (TEXT), completed_date (TEXT), UNIQUE(project_id, phase_number)

    **Update `backend/models.py`:**
    Add functions:
    - `upsert_project_state(db, project_id, parsed: ProjectFullState)` — INSERT OR REPLACE into project_state with all parsed fields, store blockers/decisions as JSON strings
    - `upsert_project_phases(db, project_id, phases: list[PhaseProgress])` — DELETE existing phases for project_id, INSERT new ones
    - `get_project_state(db, project_id) -> dict | None` — SELECT from project_state
    - `get_project_phases(db, project_id) -> list[dict]` — SELECT from project_phases

    **Update `backend/schemas.py`:**
    Add response models:
    - `ProjectStateResponse` — mirrors ProjectFullState but for API output, includes state/roadmap/project nested objects and synced_at

    **Update `backend/registry.py`:**
    Add two new endpoints:
    - `POST /api/projects/{project_id}/sync` — look up project path, call parse_project_state(), call upsert functions, update last_synced, return parsed state
    - `GET /api/projects/{project_id}/state` — return cached state from SQLite (404 if never synced)
    Both endpoints should validate project exists (404 if not found).

    - Test graceful handling of empty/missing content (returns defaults, no exceptions)
    - Test parse_project_state with a real .paul/ path (this project itself — use os.path to locate)

    Avoid: Adding new dependencies — use stdlib json for serialization.
